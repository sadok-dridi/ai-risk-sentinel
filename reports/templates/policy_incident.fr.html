1. Définition d'un Incident IA

Un incident IA est tout événement impliquant un outil d'intelligence artificielle qui :

  • Provoque une divulgation non autorisée de données (prompt leakage, réponse contaminée)
  • Génère un contenu préjudiciable (toxique, discriminatoire, frauduleux)
  • Résulte d'une manipulation malveillante (injection, jailbreak, empoisonnement)
  • Produit une décision erronée ayant un impact métier (hallucination critique)
  • Constitue une violation de données personnelles au sens du RGPD
Criticité : Le taux de vulnérabilité mesuré est de {{ vulnerability_rate }}%. La probabilité d'incident est réelle. Cette procédure doit être connue de tous les utilisateurs d'outils IA.

3. Procédure Pas-à-Pas

Étape 1 — DÉTECTION
Identifier et qualifier l'incident
  • Signalement : Tout utilisateur témoin d'un comportement anormal de l'IA le signale immédiatement au RSSI via le canal dédié (email incident-ia@ ou ticket)
  • Détection automatique : Les logs de prompts/réponses sont analysés. Les alertes DLP (détection de secrets, patterns RGPD) déclenchent un ticket automatique
  • Qualification : Le RSSI détermine en moins d'1 heure si l'événement est un incident (impact avéré) ou un événement (tentative bloquée)
Étape 2 — CONFINEMENT
Isoler le périmètre et limiter la propagation
  • Révoquer immédiatement l'accès de l'utilisateur à l'outil IA concerné
  • Si fuite de données : identifier toutes les conversations contenant les données exposées
  • Si injection/jailbreak : désactiver temporairement le modèle concerné
  • Préserver les logs (prompts, réponses, timestamps) pour investigation
Étape 3 — INVESTIGATION
Analyser les causes et l'étendue
  • Reconstituer la chronologie : prompt d'origine, réponse du modèle, actions subséquentes
  • Déterminer le vecteur : erreur utilisateur, attaque délibérée, défaillance du modèle
  • Évaluer le scope : quelles données ont été exposées ? À quel récepteur ?
  • Classifier la sévérité : Mineure, Significative, Majeure, Critique
Étape 4 — REMÉDIATION
Corriger et renforcer
  • Appliquer le correctif immédiat : renforcer le prompt système, activer un filtre, bloquer un pattern
  • Mettre à jour le registre des risques (PTR) avec les nouvelles mesures
  • Tester la correction : relancer le scénario d'attaque en environnement contrôlé
  • Si nécessaire, déployer une contre-mesure globale (ex : DLP renforcé, quota réduit)
Étape 5 — NOTIFICATION
Informer les parties prenantes
    {% if has_personal_data %}
  • CNIL : Notification obligatoire sous 72 heures si des données personnelles sont compromises (Article 33 RGPD)
    • {% endif %}
  • Personnes concernées : Si risque élevé pour les droits et libertés, notification individuelle
  • Direction : Compte-rendu au CODIR dans les 24h pour les incidents majeurs/critiques
  • Fournisseur IA : Si l'incident est lié au modèle, notification au fournisseur
Étape 6 — POST-MORTEM
Tirer les leçons et améliorer
  • Rédiger un rapport d'incident dans les 5 jours ouvrés
  • Identifier les causes racines (5 Why)
  • Mettre à jour la politique d'usage, les formations et le PTR
  • Présenter les conclusions en revue RSSI trimestrielle
